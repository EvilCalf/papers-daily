## 2. 研究目标（详细）

- **核心痛点**: 
  1. 现有基于大语言模型（LLM）的代理自我进化工作主要将成功经验记录为文本提示（textual prompts）或反思（reflections）。
  2. 文本经验在复杂场景中无法可靠保证任务的高效重新执行（efficient task re-execution）。
  3. 缺乏一种能够跨系统移植且随任务增多而愈发鲁棒的机制。
- **研究问题**: 如何构建一个能够持续积累能力、减少人工干预且保证任务执行可靠性的 LLM 代理自我进化框架？
- **研究目标**:
  - 将成功解决方案保存为纯 Python 代码形式的可执行子代理（executable subagent）。
  - 基于执行反馈持续细化子代理，使其随任务遇到次数增加而变得更鲁棒。
  - 实现子代理库的标准化文档和跨系统移植能力。
  - 证明框架能随时间推移降低类似任务所需的努力（effort）。
- **应用场景**: 复杂自动化任务流、软件开发辅助、需要长期记忆和技能积累的自主代理系统。

## 3. 方法概述（技术细节）

- **核心思想**: 摒弃传统的文本记忆模式，采用“代码即记忆”（Code as Memory）范式，通过元代理（Meta-Agent） orchestration 管理可执行子代理的生命周期。
- **架构设计**: 
  框架遵循三阶段生命周期：**Install**（安装）、**Self-Evolve**（自我进化）、**Deploy**（部署）。核心组件包括元代理编排器（Meta-Agent orchestra）和子代理池（Subagent Pool）。架构符合新兴的 Agent Skills 开放标准。
- **关键创新**:
  1. **可执行子代理积累**: 成功任务被转化为纯 Python 代码块，包含标准化文档。这与传统的向量数据库存储文本片段不同，代码具有确定性执行逻辑。
  2. **基于反馈的持续细化**: 子代理并非一次性生成，而是根据执行反馈（execution feedback）不断修正。公式化表达为 $Code_{t+1} = Refine(Code_t, Feedback_t)$，其中 $Feedback_t$ 包含运行错误、输出质量等信号。
  3. **元代理动态编排**: 元代理根据任务需求查询子代理池。如图 1 所示，若不存在相关子代理（Q1 情况），则从头创建；若存在匹配子代理（Q2 情况），则复用并修改。大多数任务处于两者之间，部分复用部分新建。
- **与现有方法的区别**: 
  现有 Self-Evolving Agents 依赖文本经验（Textual Experience），存在语义模糊和执行不确定性；AgentFactory 依赖可执行代码，具备类型检查和环境隔离优势，且支持跨 Python 系统移植。

## 4. 数据和评估（完整信息）

- **数据集**: 
  - **名称**: 自建任务基准（Custom Task Benchmark）。
  - **规模**: Batch 1 包含 15 个真实世界任务（15 real-world tasks）；Batch 2 为后续相似任务集，用于测试复用能力。
  - **来源**: 人工策划的真实场景任务。
  - **预处理**: 任务被标准化为元代理可理解的输入格式。
- **评估设置**:
  - **对比方法**: 
    1. **ReAct**: 经典的推理与行动框架。
    2. **Self-Evolving Agents (Textual)**: 基于文本经验积累的基线。
    3. **AgentFactory**: 本文提出的基于代码积累的方法。
  - **评估指标**: **Avg. Tokens / Task**（每个任务平均输出 Token 数）。仅计算编排代理（orchestrating agent）的消耗，不包含子代理内部 LLM 消耗。越低表示效率越高。
  - **实验配置**: 使用两种 LLM 骨干模型：**Opus 4.6** 和 **Sonnet 4.6**。
- **实现细节**:
  - **框架**: 基于 Python 实现的代理框架。
  - **代码开源**: 完整实现已开源。
  - **训练/进化**: 无需传统梯度下降训练，通过交互反馈进行代码迭代进化。

## 5. 关键结果（包含具体数字）

- **主实验结果**:

| Method | Task Setting | Opus 4.6 (Tokens) | Sonnet 4.6 (Tokens) |
| :--- | :--- | :--- | :--- |
| **ReAct** | Batch 1 | 8298 | 6893 |
| **ReAct** | Batch 2 | 7022 | 7029 |
| **Self-Evolving (Text)** | Batch 1 (from scratch) | 8608 | 8163 |
| **Self-Evolving (Text)** | Batch 2 (w/ saved) | 6210 | 8223 |
| **AgentFactory** | Batch 1 (from scratch) | 4324 | 9199 |
| **AgentFactory** | Batch 2 (w/ saved) | **2971** | **3862** |

- **关键发现**:
  1. **显著的效率提升**: 在 Opus 4.6 模型上，AgentFactory 在 Batch 2 的 Token 消耗（2971）相比 Batch 1（4324）降低了约 **31.3%**，相比 ReAct Batch 2（7022）降低了 **57.7%**。
  2. **复用优势明显**: 相比基于文本的自我进化代理（Opus Batch 2 为 6210），AgentFactory 进一步减少了 **52.1%** 的编排 Token 消耗，证明代码复用比文本检索更高效。
  3. **模型差异性**: 在 Sonnet 4.6 上，AgentFactory 初始成本较高（9199），但复用后大幅下降至 3862，降幅达 **58.0%**，表明初期代码生成开销大，但长期收益极高。
  4. **稳定性**: ReAct 和文本基线在 Batch 2 中表现波动较大（如 Sonnet 下文本基线反而上升），而 AgentFactory  consistently 下降。
- **消融实验**: 论文通过 Batch 1（无保存）与 Batch 2（有保存）的对比，隐含了关于“子代理复用机制”的消融验证，证明保存机制是性能提升的关键。
- **定性结果**: 演示视频展示了子代理库随时间增长，处理类似任务时元代理的决策路径显著缩短。

## 6. 优势（量化对比）

- **性能优势**: 相比 SOTA 方法（ReAct），在复杂任务复用场景下（Batch 2），编排 Token 消耗降低超过 **50%**（Opus 下从 7022 降至 2971）。
- **效率优势**: 随着任务数量增加，边际成本递减。文本基线在 Sonnet 模型上甚至出现效率倒退（8163 -> 8223），而 AgentFactory 始终保持优化趋势。
- **通用性优势**: 保存的子代理为纯 Python 代码，具备标准化文档，理论上可移植至任何支持 Python 的系统，不依赖特定 LLM 提供商的私有格式。
- **理论优势**: 将非结构化的“经验”转化为结构化的“代码”，利用了编程语言的确定性特性，解决了文本提示在复杂逻辑执行中的不可靠性问题。

## 7. 局限性和风险（诚实评估）

- **技术局限**: 
  1. **初始成本高**: 在 Sonnet 4.6 上，Batch 1 的 Token 消耗（9199）远高于 ReAct（6893），表明构建可执行代码的初期开销巨大，不适合一次性任务。
  2. **代码执行风险**: 动态生成并执行 Python 代码存在安全漏洞（如无限循环、恶意代码注入），需要严格的沙箱环境。
- **实验局限**: 
  1. **任务规模**: 仅使用了 15 个真实世界任务（Batch 1），样本量较小，可能不足以代表所有复杂场景。
  2. **模型依赖**: 实验仅基于 Opus 4.6 和 Sonnet 4.6，未验证在开源小模型上的表现。
- **应用风险**: 依赖管理（Dependency Management）是主要挑战，子代理代码可能依赖特定库，导致跨系统移植时出现环境冲突。
- **未来改进方向**: 引入自动化测试生成以确保子代理代码质量；优化初始代码生成策略以降低 Batch 1 开销；扩展支持非 Python 环境。

## 8. 可重复性说明

- **代码可用性**: **完全开源**。提供了 GitHub 仓库（https://github.com/zzatpku/AgentFactory），包含核心框架实现。
- **数据可用性**: 任务集未明确说明是否公开下载，但描述了任务设计逻辑（15 个真实任务），研究者可根据描述复现类似基准。
- **预训练模型**: 未提供预训练子代理库，用户需从零开始进化，或自行积累。
- **复现难度**: **中等**。需要配置 LLM API 接口及 Python 沙箱执行环境。核心难点在于复现元代理的编排逻辑和代码细化反馈机制。

## 9. 实践启示

- **对研究者的价值**: 提供了“代码即记忆”的新范式，启示后续研究可探索将 Agent 的记忆模块从向量数据库转向代码仓库。数学上可进一步建模代码演化的收敛性。
- **对工程师的价值**: 在实际落地中，应优先考虑将高频任务固化为函数或微服务，而非依赖 Prompt 优化。需注意沙箱安全性设计。
- **对行业的价值**: 为构建长期运行的自主代理（Autonomous Agents）提供了基础设施思路，有助于降低企业级自动化任务的长期运营成本（Token 成本）。
- **跟进建议**: 推荐阅读关于"Program Synthesis with LLMs"及"Agent Memory Mechanisms"的相关论文，关注 Agent Skills 开放标准的后续发展。

## 10. 简要结论（精华总结）

AgentFactory 提出了一种基于可执行子代理代码积累的自我进化框架，核心创新在于用结构化代码替代文本经验作为记忆载体。实验数据显示，在任务复用场景下，该方法相比 ReAct 基线将编排 Token 消耗降低了超过 50%（Opus 模型下从 7022 降至 2971），显著提升了效率。尽管初期构建成本较高，但其带来的长期能力积累和跨系统移植性，为构建鲁棒、高效的 LLM 代理系统提供了重要的技术路径和实践参考。
