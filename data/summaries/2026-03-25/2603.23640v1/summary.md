## 2. 研究目标（详细）

- **核心痛点**:
    1.  **持续推理的热约束**: 现有研究多关注峰值性能，忽视了“始终在线”个人代理场景下，硬件因热积累导致的性能衰减（Thermal Throttling）。
    2.  **能效比评估缺失**: 缺乏在不同功耗包络（Power Envelope）下，不同架构（NPU vs GPU vs Mobile SoC）的能量比例性（Energy Proportionality）对比。
    3.  **系统级行为不可预测**: 移动操作系统（OS）的强制频率限制策略对推理任务的中断影响未被充分量化。
- **研究问题**: 在功率、热包络和内存严格受限的边缘硬件上，部署量化大语言模型进行持续推理时，吞吐量、延迟、功耗和热行为的具体权衡关系是什么？
- **研究目标**:
    - 量化四类主流边缘平台（树莓派 +NPU、安卓旗舰、iOS 旗舰、笔记本 GPU）在 20 次连续迭代下的性能稳定性。
    - 识别不同硬件架构在持续负载下的主要瓶颈（是热管理、内存带宽还是功耗上限）。
    - 建立能量比例性基准，评估低功耗硬件是否能通过效率弥补吞吐量的不足。
    - 为边缘 AI 代理的系统架构设计提供硬件选型依据。
- **应用场景**: 始终在线的个人智能助理、离线语音交互系统、嵌入式边缘计算节点、隐私敏感型本地化 LLM 服务。

## 3. 方法概述（技术细节）

- **核心思想**: 采用控制变量法，在统一模型（Qwen 2.5 1.5B 4-bit）和统一负载（258-token）下，通过长时间持续推理测试，捕捉硬件从冷启动到热饱和状态的性能曲线。
- **架构设计**:
    - **被测模型**: Qwen 2.5 1.5B。基于 Transformer 架构，包含约 15 亿参数。采用 4-bit 量化（INT4），显著降低内存占用和计算精度需求。架构包含多层自注意力机制（Self-Attention）和前馈神经网络（FFN），激活函数通常为 SwiGLU。
    - **量化技术**: 使用 4-bit 权重量化，可能结合 KV Cache 量化。数学上，权重 $W$ 从 FP16 映射到 INT4 空间，$W_{int4} = \text{round}(W_{fp16} / \text{scale})$，推理时进行反量化计算或整型矩阵乘法。
- **关键创新**:
    - **热饱和测试协议**: 不同于单次推理测试，本方法执行 20 次“暖条件”迭代（Warm-condition Iterations），强制设备进入热稳定状态，模拟真实长时间运行场景。
    - **多维度指标同步采集**: 同时记录吞吐量（tok/s）、延迟、实时功耗（W）和热行为，而非单一性能指标。
    - **跨架构对比框架**: 将离散 NPU（Hailo）、集成移动 SoC（Snapdragon/Apple Silicon）和独立 GPU（NVIDIA）置于同一评估框架下，消除软件栈差异带来的部分偏差。
    - **能量比例性分析**: 引入能量比例性指标，计算每瓦特产生的吞吐量，评估硬件在低负载下的效率表现。
- **与现有方法的区别**: 现有基准（如 MLPerf）多关注峰值吞吐量或单次推理延迟，忽略了热节流导致的性能下降曲线；本研究专注于“持续负载（Sustained Load）”下的系统行为，揭示了 OS 级电源管理对 AI 任务的干扰。

## 4. 数据和评估（完整信息）

- **数据集**:
    - **名称**: 固定基准提示词（Fixed Benchmark Prompt）。
    - **规模**: 单个提示词，长度为 258 tokens。
    - **数据来源**: 人工构造或选自标准基准集，确保内容复杂度一致，排除生成内容随机性对推理时间的干扰。
    - **预处理方式**: 统一进行 4-bit 量化处理，输入文本标准化为模型接受的 Token ID 序列。
- **评估设置**:
    - **对比方法（硬件平台）**:
        1.  Raspberry Pi 5 + Hailo-10H NPU
        2.  Samsung Galaxy S24 Ultra (Mobile SoC)
        3.  iPhone 16 Pro (Mobile SoC)
        4.  Laptop NVIDIA RTX 4050 GPU
    - **评估指标**:
        - **吞吐量 (Throughput)**: tokens per second (tok/s)，衡量生成速度。
        - **功耗 (Power)**: Watts (W)，衡量能源消耗。
        - **热行为 (Thermal Behaviour)**: 温度变化及导致的频率调整。
        - **稳定性 (Stability)**: 20 次迭代内的性能方差。
    - **实验配置**: 每设备执行 20 次连续迭代，初始状态为“暖条件”（模拟设备已有一定基础温度）。
- **实现细节**:
    - **推理引擎**: 未明确指定，通常涉及 llama.cpp、MLC LLM 或厂商专用推理引擎（如 CoreML、SNPE）。
    - **量化格式**: 4-bit quantised (INT4)。
    - **测量工具**: 外部功率计或硬件内部传感器（如 NVIDIA NVML, Android Battery Stats）。

## 5. 关键结果（包含具体数字）

- **主实验结果**:

| 硬件平台 | 持续吞吐量 (tok/s) | 持续功耗 (W) | 热/功耗约束表现 | 能量比例性 (tok/W) |
| :--- | :--- | :--- | :--- | :--- |
| **NVIDIA RTX 4050** | **131.7** | 34.1 | 受电池功耗上限限制 | ~3.86 |
| **Hailo-10H NPU** | 6.9 | < 2.0 | 受模块内存带宽限制 | ~3.45 |
| **iPhone 16 Pro** | 初始值约 50% 后衰减 | 未详述 | 2 次迭代内吞吐量损失近半 | 低 |
| **Samsung S24 Ultra** | 推理终止 | 未详述 | OS 强制 GPU 频率底限导致中断 | 0 |

- **关键发现**:
    - **移动平台热管理主导**: 对于 iPhone 16 Pro，热管理取代峰值计算成为主要约束，仅在 2 次迭代内吞吐量损失接近 50%，表明移动 SoC 无法维持高频推理。
    - **极端节流现象**: Samsung S24 Ultra 遭遇硬性 OS 强制 GPU 频率底限，导致推理任务完全终止，揭示了安卓电源管理策略对持续 AI 负载的激进限制。
    - **能效匹配**: Hailo-10H 虽吞吐量低（6.9 tok/s），但在能量比例性上与 RTX 4050 匹配，且在功耗低于 2W 的情况下运行，方差接近零。
    - **吞吐量差异**: RTX 4050 的吞吐量是 Hailo-10H 的 19 倍（131.7 vs 6.9），但功耗高出 17 倍以上。
- **消融实验**: 论文未明确提及传统意义上的消融实验，但通过对比不同硬件架构（NPU vs GPU vs SoC）实现了架构层面的消融分析，隔离了计算单元类型对性能的影响。
- **定性结果**: 移动设备在持续负载下表现出明显的性能不稳定曲线，而专用 NPU 表现出极高的稳定性（近零方差），适合对延迟抖动敏感的应用。

## 6. 优势（量化对比）

- **性能优势**:
    - **稳定性**: Hailo-10H NPU 在 20 次迭代中表现出“近零方差（near-zero variance）”的吞吐量，相比移动手机性能波动超过 50% 的情况，稳定性提升显著。
    - **持续能力**: 在移动设备因过热降频或终止任务时，专用硬件（RTX 4050, Hailo）能维持全程推理不中断。
- **效率优势**:
    - **低功耗运行**: Hailo-10H 在低于 2W 的功耗下运行，相比 RTX 4050 的 34.1W，功耗降低约 94%。
    - **能量比例性**: 尽管吞吐量低 19 倍，但 Hailo-10H 的能量比例性（tok/W）与高性能 GPU 相当，证明其在电池供电场景下的极端效率优势。
- **通用性优势**:
    - **场景覆盖**: 研究覆盖了从嵌入式（Pi）、移动端（手机）到边缘计算端（笔记本）的全谱系设备，结论具有广泛的硬件代表性。
    - **模型兼容性**: 基于 Qwen 2.5 1.5B 的测试结果可类推至同量级（1B-3B）的其他量化模型。
- **理论优势**:
    - **系统级视角**: 突破了单纯比较算力（TOPS）的传统视角，引入了热包络和 OS 策略作为核心变量，更贴近实际部署理论。

## 7. 局限性和风险（诚实评估）

- **技术局限**:
    - **模型单一性**: 仅测试了 Qwen 2.5 1.5B 单一模型，未涵盖更大参数（如 7B、70B）或不同架构（如 Mamba）的表现，结论可能无法外推至所有 LLM。
    - **负载单一性**: 仅使用 258-token 固定提示词，未测试长上下文（Long Context）或复杂推理任务对内存带宽的极端压力。
- **实验局限**:
    - **环境温度未控**: 摘要未提及实验是否在恒温箱中进行，环境温度波动可能影响热节流阈值。
    - **软件栈差异**: 不同设备使用的推理后端（如 CoreML vs TensorRT vs TFLite）优化程度不同，可能混淆硬件性能差异。
- **应用风险**:
    - **手机部署风险**: 依赖手机 GPU 进行持续推理存在任务被 OS 杀死的风险（如 S24 Ultra 案例），不适合关键任务。
    - **NPU 带宽瓶颈**: 专用 NPU 受限于片上内存带宽，难以扩展到更大模型，存在扩展性风险。
- **未来改进方向**:
    - 增加更多模型尺寸和量化精度（如 8-bit, FP8）的对比。
    - 引入主动散热方案（如风扇）测试对移动设备性能恢复的影响。
    - 深入研究 OS 电源管理策略的可配置性，以解锁更多性能。

## 8. 可重复性说明

## 9. 实践启示

- **对研究者的价值**:
    - 揭示了“峰值算力”在边缘侧的误导性，建议后续研究将“持续吞吐量”和“热稳定时间”作为核心指标。
    - 提供了边缘 LLM 能效评估的新范式，即能量比例性比绝对速度更重要。
- **对工程师的价值**:
    - **硬件选型**: 若应用对功耗敏感且吞吐量要求低（<10 tok/s），专用 NPU（如 Hailo）是最佳选择；若需高吞吐且可接受高功耗，选择离散 GPU。
    - **避免手机依赖**: 关键业务逻辑不应完全依赖手机 GPU 进行持续推理，需设计降级策略以防 OS 终止进程。
    - **散热设计**: 边缘设备部署需考虑被动散热片或主动风扇，以延缓热节流发生。
- **对行业的价值**:
    - 推动芯片厂商优化 NPU 内存带宽，而非单纯增加算力。
    - 促使移动操作系统厂商开放更细粒度的电源管理策略给 AI 应用，避免误杀持续推理进程。
- **跟进建议**:
    - 推荐阅读关于 LLM 量化技术（如 AWQ, GPTQ）的论文，以进一步优化边缘内存占用。
    - 关注 MLPerf Inference Edge 基准的最新更新，对比本研究结果。

## 10. 简要结论（精华总结）

本研究通过对比四类边缘硬件平台，揭示了大语言模型在持续负载下的真实性能瓶颈。核心贡献在于量化了热管理和操作系统策略对移动设备推理稳定性的致命影响，以及专用 NPU 在能效比上的优势。关键结果显示，RTX 4050 虽提供 131.7 tok/s 的高吞吐，但功耗高达 34.1W；而 Hailo-10H 以低于 2W 的功耗实现了相当的能量比例性，尽管吞吐量低 19 倍。实际意义在于指导开发者根据功耗包络和稳定性需求，而非峰值算力，来选择边缘 AI 部署硬件。
